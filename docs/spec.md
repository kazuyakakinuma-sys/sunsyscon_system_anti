# sunsyscon — システム仕様書

## 1. プロジェクト概要

楽天銀行の入金履歴CSVと、kintone各アプリからエクスポートしたCSVを照合・処理し、  
kintoneへ一括インポートできる更新用CSVを自動生成するWebアプリ。  
kintoneとの連携はすべてCSVの入出力で完結し、APIは使用しない。

---

## 2. 背景・課題

- 楽天銀行の入金データとkintone上の顧客データを手動で照合・転記している
- 複数のkintoneアプリにまたがる更新が発生し、作業工数が大きい
- 本ツールで照合・算出・CSV生成を自動化し、kintoneへのCSVアップロードのみで完結させる

---

## 3. システムフロー（全体）

```
【Phase 01】楽天銀行CSVをアップロードし、振込名義等の正規化処理を行う
          ↓
【STEP 2】kintone各アプリからエクスポートしたCSVをアップロード
          （複数ファイル）
          ↓
【STEP 4〜N】アプリ内で照合・計算処理（複数ステップ）
          ↓
【STEP 最終】kintoneインポート用CSVを生成・ダウンロード
          ↓
   kintoneにCSVインポートで一括更新 ✅
```

## 4. 入力データ

### 4.1 入力① ：楽天銀行CSV（Phase 01）

| 列 | 列名 | 内容 | 例 |
|----|------|------|----||
| A | 取引日 | 入出金の取引日付（YYYYMMDD形式） | 20260219 |
| B | 入出金(円) | 入金はプラス、出金はマイナス | 5680 |
| C | 累計(円) | 取引後の残高累計 | 672743671 |
| D | 入出金先内容（振込名義） | 振込人名（全角カタカナ）または出金先情報 | イツ゛マ　ヒロユキ |

**ファイル仕様**
- 文字コード：**Shift-JIS**（読み込み時にUTF-8変換が必要）
- 区切り文字：カンマ区切り（CSV）
- ヘッダー行：1行目（`取引日,入出金(円),入出金先内容`）

**実データの特徴（サンプルより）**
- 入金行：`20260219,5680,イツ゛マ　ヒロユキ` のような形式
- 出金行：銀行名・支店・口座番号・振込先情報が長文で含まれる
- 振込手数料行：`振込手数料（内消費税 ￥20）` のような形式
- 半濁点（゛）が文字と独立して存在する場合あり（例：`イツ゛マ` = `イヅマ`）
- 全角スペースで姓・名が区切られている場合あり

- 処理対象：**B列がプラス（入金）のレコードのみ**

## 5. 中間処理ステップ

### Phase 01：楽天銀行CSVの正規化・変換

楽天銀行CSVの振込名義（C列）は、銀行システムの制約でカタカナ表記が崩れているため、  
kintoneの顧客フリガナと照合できるよう正規化する。

#### 変換ルール

**① 除外パターン判定（先頭処理）**
- 以下のパターンを含む行はフリガナを**空欄**として扱い、以降の変換をスキップする

| パターン | 対象 |
|---------|------|
| `依頼人名：` | 自社出金・振替行 |
| `ＣＳＳ（ＭＦＲＬ` | CSS（MFRL賃料）管理費振込 |
| `ロボツトペイメント` | ロボットペイメント（決済代行会社） |
| `コクリツケンキユウカイハツ` | 国立研究開発法人医薬基盤健康 |
| `ラクテンカ−ト゛` | 楽天カードサービス |
| `ニホンセイメイホケン` | 日本生命保険 |

> ※ 上記法人・サービスからの**入金であっても**照合対象外として空欄にする（業務上の仕様）。

**② 不要文字の削除**
- 漢字・数字・記号・英語・ハイフンをすべて削除
- 削除後、文字が残らなかった行（例：`振込手数料`の行）は**削除せず順番を維持**する

**② 半濁点の結合**
- 独立した半濁点（゛）を直前の文字と結合する
  - 例：`イツ゛マ` → `イヅマ`
  - 例：`オカ゛ワ` → `オガワ`

**③ 拗音（小文字）への変換**

| 変換前 | 変換後 | 変換前 | 変換後 | 変換前 | 変換後 |
|--------|--------|--------|--------|--------|--------|
| キヤ | キャ | キユ | キュ | キヨ | キョ |
| シヤ | シャ | シユ | シュ | シヨ | ショ |
| チヤ | チャ | チユ | チュ | チヨ | チョ |
| ニヤ | ニャ | ニユ | ニュ | ニヨ | ニョ |
| ヒヤ | ヒャ | ヒユ | ヒュ | ヒヨ | ヒョ |
| ミヤ | ミャ | ミユ | ミュ | ミヨ | ミョ |
| リヤ | リャ | リユ | リュ | リヨ | リョ |
| ギヤ | ギャ | ギユ | ギュ | ギヨ | ギョ |
| ジヤ | ジャ | ジユ | ジュ | ジヨ | ジョ |
| ビヤ | ビャ | ビユ | ビュ | ビヨ | ビョ |
| ピヤ | ピャ | ピユ | ピュ | ピヨ | ピョ |

**④ 促音（ッ）への変換**
- `ツ` + 子音カタカナ の組み合わせを `ッ` + 子音カタカナ に変換
  - 例：`チヨツキ` → まず `チョツキ` → `チョッキ`

**⑤ 例外処理（変換しない名前）**
- 以下のような、大きい「ユ」「ヨ」が名前として一般的なケースは変換しない
  - 例：`ミユ`、`ミユキ`、`ユウイチ`、`ヨウイチ` など
- 日本の氏名・苗字として自然な場合は変換を適用する

**⑥ 変換後の整形**
- 全角スペースを除去してフリガナとして整形

#### 変換例（サンプルCSVより）

| 変換前（銀行CSV） | 変換後（正規化フリガナ） | 備考 |
|-----------------|----------------------|------|
| イツ゛マ　ヒロユキ | イヅマヒロユキ | 半濁点結合 |
| ナカジヨウ　マサノリ | ナカジョウマサノリ | ジヨ→ジョ変換 |
| サイトウ　シヨウタ | サイトウショウタ | シヨ→ショ変換 |
| サトウ　シユウ | サトウシュウ | シユ→シュ変換 |
| ナガトモ　シユンヤ | ナガトモシュンヤ | シユ→シュ変換 |
| アカハネ　リヨウスケ | アカハネリョウスケ | リヨ→リョ変換 |
| ルセロ　ゲリコ　デイアス | （空文字） | 外国人名→削除対象なし・順番維持 |
| 振込手数料（内消費税 ￥20） | （空文字） | 文字削除後→削除せず順番維持 |

### STEP 4：顧客データの照合と計算（Phase 3）

「顧客データアプリ」のCSV（※顧客データ元）と楽天銀行CSVのデータを照合し、過不足や遅延状態の計算を行う。

**照合ロジック（VLOOKUPの代替）**
- 楽天銀行の振込名義（STEP 3で正規化済みフリガナ）をキーに、顧客データCSVの `フリガナ` を突合する。
- 突合できた場合、顧客データから以下の情報を引き当てる：
  `レコード番号`、`顧客ID`、`氏名`、`契約月数`、`合計金額`(月額)、`過不足`(過去分)、`支払ステータス`、`契約状況ステータス`、`遅延認証`

**料金・過不足の算出ロジック**
- **リアルタイム料金計算** = `顧客データCSVの「合計金額」` - `顧客データCSVの「過不足」`
- **手数料の加減算ルール**（本来の「支払方法」と異なる手段で入金があった場合の請求額調整）：
  - `顧客データの支払方法` が `口座振替` 等で、今回（銀行振込で）入金があった場合： `リアルタイム料金計算` **＋ 1000円** を正しい請求額とする。
  - `顧客データの支払方法` が `コンビニ払い` 等で、今回（銀行振込で）入金があった場合： `リアルタイム料金計算` **− 330円** を正しい請求額とする。
  - `顧客データの支払方法` が `銀行振込` で、そのまま（銀行振込で）入金があった場合： `リアルタイム料金計算` をそのまま正しい請求額とする。
- **過不足計算（今回発生分）** = `楽天銀行CSVの「入金額」` - `( 手数料調整後のリアルタイム料金計算 )`
- **契約月数（更新後）** = `顧客データCSVの「契約月数」` + 1
- **遅延回数（更新後）** = `遅延情報の「遅延回数」` + 1

### STEP 5：処理結果の振り分け（Phase 3後半）

計算結果から、対象レコードを以下の条件で各ステータスに分け、Web上のUIで一覧表示する。

1. **入金済み（完全一致）**: 氏名マッチングが1件のみヒットし、過不足が0で正しく入金されたデータ。自動でインポート対象となる。
2. **要確認（複数ヒット・金額不一致・NA）**: 
  - 氏名マッチングが複数人（同姓同名など）ヒットした場合。
  - 照合先が見つからない（NA）、または入金額と請求額が合わない（過不足発生）場合。
  ▶ **※これらはWeb UI上でオペレーターが手動で照合する（後述のUI機能）。**

---

### STEP 5.5：オペレーターによる手動チェックと紐付け機能（Web UI）

「要確認」に振り分けられたデータについて、UI上で以下の操作を行えるようにする。
- 複数ヒットの場合、候補となる顧客のリスト（顧客ID、氏名、振込金額、請求額、過不足など）を表示。
- オペレーターが正しい顧客を選択・紐付けすると、そのデータは「入金済み」としてインポート対象に送られる。
- 金額不一致（過不足）のまま許可するかどうかなどの判断もここで行う。

---

### STEP 6：入金済み顧客のインポートCSV生成（Phase 4）

STEP 5で「入金済み者リスト」に振り分けられたデータをもとに、kintoneへ一括登録するための最終CSVを出力する。
この生成処理はExcel手作業の「値貼り付け・列削除」を自動化するものである。

## 6. 出力データ（kintoneインポート用CSV）

### 6.1 出力CSVフォーマット

| 列 | 列名 | 内容・算出ロジック |
|----|------|--------------------|
| A | レコード番号 | 顧客データ側から取得した `レコード番号` |
| B | 顧客ID | 顧客データ側から取得した `顧客ID` |
| C | 氏名 | 顧客データ側から取得した `氏名` |
| D | フリガナ | 顧客データ側から取得した `フリガナ` |
| E | 契約月数 | 取得した月数 + 1 （STEP 4で算出） |
| F | 過不足 | 入金額と請求額の差分計算結果（STEP 4で算出） |
| G | 支払ステータス | **固定文字列** `入金済み` をセット |
| H | 契約状況ステータス | **固定文字列** `継続契約` をセット |
| I | 遅延認証 | 顧客データから取得した `遅延認証`（内容に変更がないか確認） |

```csv
レコード番号,顧客ID,氏名,フリガナ,契約月数,過不足,支払ステータス,契約状況ステータス,遅延認証
1001,C-0001,山田太郎,ヤマダタロウ,13,0,入金済み,継続契約,-
1002,C-0002,鈴木花子,スズキハナコ,7,-5000,入金済み,継続契約,あり
```
